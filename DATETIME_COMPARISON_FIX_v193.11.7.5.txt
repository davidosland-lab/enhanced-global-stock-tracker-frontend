BUG DESCRIPTION
================================================================================

**Symptom:**
Trading cycle crashes with TypeError when checking for updated morning reports:
```
TypeError: can't compare offset-naive and offset-aware datetimes
```

**Location:**
- File: `core/paper_trading_coordinator.py`
- Method: `_check_for_updated_reports()`
- Line: 540

**Root Cause:**
Comparing timezone-aware datetime (from ISO format string) with timezone-naive 
datetime (from file timestamp) without normalizing both to same timezone state.

**Impact:**
- Trading cycle crashes when AU morning report is updated
- System recovers but skips that cycle
- Happens ~once per day when report files are refreshed

**Error Log Example:**
```

Traceback (most recent call last):

       ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

2026-03-12 13:35:16,073 - paper_trading_coordinator - WARNING - [WARN]  Trading cycle failed but loop will continue
```

================================================================================
THE FIX
================================================================================

**Solution:**
Normalize both datetimes to naive (timezone-less) before comparison by stripping 
timezone info if present.

**Code Change:**
```python
# BEFORE (BROKEN):
file_mtime = datetime.fromtimestamp(report_path.stat().st_mtime)  # naive
cached_time = datetime.fromisoformat(cached_report['timestamp'])  # may be aware
if file_mtime > cached_time:  # ❌ CRASH if cached_time is aware

# AFTER (FIXED):
file_mtime = datetime.fromtimestamp(report_path.stat().st_mtime)  # naive
cached_time = datetime.fromisoformat(cached_report['timestamp'])
# Remove timezone info from both for comparison
if cached_time.tzinfo is not None:
    cached_time = cached_time.replace(tzinfo=None)  # normalize to naive
if file_mtime > cached_time:  # ✅ Both naive, comparison works
```

ADDITIONAL CONTEXT
================================================================================

**Why This Bug Occurred:**
- `datetime.fromtimestamp()` returns naive datetime (no timezone)
- JSON report files may store timestamps in ISO format with timezone (e.g., "2026-03-12T08:30:00+11:00")
- `datetime.fromisoformat()` preserves timezone if present in string
- Python 3.x raises TypeError when comparing aware vs naive datetimes

**Best Practice Applied:**
- Always normalize datetimes to same timezone state before comparison
- Either both aware OR both naive
- For file timestamp comparisons, naive is simpler (no timezone math needed)

**Related Python Documentation:**
https://docs.python.org/3/library/datetime.html#aware-and-naive-objects
