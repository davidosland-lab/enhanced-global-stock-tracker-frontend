================================================================================

USER'S REPORTED ISSUE:
----------------------
• Positions appearing/disappearing randomly in dashboard
• NAB.AX: 277 shares → disappeared → 225 shares (different quantity!)
• JUP.L: 85 shares → 98 shares (increased without trade)
• CNA.L: 65 shares → 75 shares (increased without trade)
• Total Trades: 0 (positions changing but no trades recorded)
• No "[OK] POSITION CLOSED" messages in logs

STATE FILE EVIDENCE:
--------------------
From user's logs showing file sizes:
  16:16:08: 3609 bytes (NAB.AX 277, JUP.L 85, CNA.L 65)
  16:16:44: 3581 bytes (JUP.L 98, CNA.L 75, SSE.L 4) ← NAB.AX gone!
  16:17:22: 2737 bytes (JUP.L 98, NAB.AX 225) ← Different positions!

ROOT CAUSE FOUND:
-----------------
**POSITION OVERWRITE BUG** in enter_position() function

Code Analysis (paper_trading_coordinator.py):

Line 2130:
    self.positions[symbol] = position  ← OVERWRITES existing position!

Problem Flow:
1. Pipeline recommendations call enter_position() with keyword args
2. Regular scan also calls enter_position() for same symbols  
3. If symbol already in self.positions, it OVERWRITES instead of SKIPPING
4. Old position object is destroyed without proper exit
5. No trade record created (Total Trades stays 0)
6. No capital adjustment for the "lost" position
7. New position created with different share count

Example of Bug in Action:
-------------------------
CYCLE 1 (16:16:08):
  - JUP.L position created: 85 shares @ $189.60
  - self.positions['JUP.L'] = Position(85 shares)

CYCLE 2 (16:16:44):
  - Pipeline recommendations finds JUP.L again
  - Checks: if 'JUP.L' in self.positions: continue  ← SHOULD SKIP!
  - BUT: Different code path doesn't check properly
  - Calls enter_position('JUP.L', signal)
  - Line 2130: self.positions['JUP.L'] = Position(98 shares)  ← OVERWRITE!
  - Old Position(85 shares) is lost forever
  - No exit_position() called
  - No trade recorded
  - Capital was deducted for 85 shares but now tracking 98 shares

EVIDENCE FROM CODE:
-------------------
1. enter_position() function (line 2010):
   - No check for existing position before creating new one
   - Directly assigns: self.positions[symbol] = position

2. exit_position() function (line 2411):
   - Increments self.metrics['total_trades'] (line 2433)
   - Logs "[OK] POSITION CLOSED" (line 2485)
   - User's logs show NEITHER of these!

3. Multiple entry points call enter_position():
   - Regular scan (line ~2654)
   - Pipeline recommendations (line 763)
   - Manual trades from dashboard
   - Pre-market gap trading (line ~1096)

4. Race conditions possible:
   - Dashboard updates running in separate thread
   - Trading cycle running in main thread
   - Both could call enter_position() for same symbol

THE FIX NEEDED:
---------------
1. ADD GUARD CHECK in enter_position():
   ```python
   def enter_position(self, symbol: str, signal: Dict) -> bool:
       try:
           # CRITICAL: Check if position already exists
           if symbol in self.positions:
               logger.warning(f"{symbol}: TRADE BLOCKED - Position already exists")
               logger.info(f"  Existing: {self.positions[symbol].shares} shares @ USD{self.positions[symbol].entry_price:.2f}")
               logger.info(f"  Attempted: New entry would overwrite existing position")
               return False
           
           # Rest of function...
   ```

2. ENSURE ALL ENTRY POINTS CHECK FIRST:
   - _process_pipeline_recommendations (line 729): Already has check ✓
   - Regular scan loop (line 2647): Already has check ✓
   - Need to add check IN enter_position() as final safety net

3. ADD LOGGING for duplicate attempts:
   - Log when duplicate entry is blocked
   - Track how many times this occurs
   - Helps diagnose why duplicate entries are attempted

WHY THIS WASN'T CAUGHT EARLIER:
--------------------------------
• The existing checks (line 729, 2647) use: if symbol in self.positions: continue
• These work correctly in normal flow
• BUT: Multiple concurrent threads/processes can bypass this
• OR: Logic error in one of the many entry points
• The final assignment (line 2130) has NO safety check
