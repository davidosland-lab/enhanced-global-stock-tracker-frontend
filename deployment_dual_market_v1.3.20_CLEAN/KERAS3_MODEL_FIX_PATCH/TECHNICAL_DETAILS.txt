KERAS VERSION HISTORY & BREAKING CHANGES
=========================================

Keras 2.x (TensorFlow <= 2.15.0):
---------------------------------
- Default save format: HDF5 (.h5)
- API: model.save('model.h5')
- Behavior: Automatically saves in HDF5 format
- File structure: Single .h5 file containing weights, architecture, optimizer
- Compatibility: Works with TensorFlow 2.x SavedModel format

Keras 3.x (TensorFlow >= 2.16.0):
---------------------------------
- Default save format: .keras (new native format)
- API change: model.save('model.h5') requires save_format parameter
- New behavior: 
  * model.save('model.h5') - MAY FAIL or use .keras format
  * model.save('model.h5', save_format='h5') - Explicit HDF5 save
  * model.save('model.keras') - Native Keras 3 format (recommended)
- Breaking change: Silent failures when saving to .h5 without format parameter
- Compatibility: Better cross-framework support (PyTorch, JAX, TensorFlow)

Fixed code:
----------
def save_model(self):
    """Save model and scaler to disk"""
    if self.model and self.is_trained:
        try:
            # Save model with Keras 3 compatibility
            # Keras 3.x requires explicit save_format='h5' for .h5 files
            try:
                # Try Keras 3 save format first
                self.model.save(self.model_path, save_format='h5')
                logger.info(f"Model saved to {self.model_path} (Keras 3 format)")
            except Exception as keras_error:
                # Fallback for older Keras 2.x
                logger.warning(f"Keras 3 save failed, trying legacy format: {keras_error}")
                self.model.save(self.model_path)
                logger.info(f"Model saved to {self.model_path} (legacy format)")
            
            # Save scaler
            with open(self.scaler_path, 'wb') as f:
                pickle.dump(self.scaler, f)
            
            return True
        except Exception as e:
            logger.error(f"Error saving model: {e}")
            import traceback
            logger.error(traceback.format_exc())
            return False
    return False


KEY CHANGES
===========

1. Added save_format='h5' parameter:
   - Primary fix for Keras 3 compatibility
   - Explicitly tells Keras to use HDF5 format
   - Required for .h5 file extension in Keras 3

2. Added nested try-except:
   - Inner try: Attempt Keras 3 save with format parameter
   - Inner except: Fallback to Keras 2 save method
   - Maintains backward compatibility with Keras 2.x

3. Enhanced logging:
   - Log which save method succeeded
   - "Keras 3 format" - save_format='h5' worked
   - "legacy format" - fallback to Keras 2 method worked
   - Added traceback logging for debugging

4. Better error handling:
   - Captures specific Keras save errors
   - Logs full traceback for debugging
   - Allows graceful fallback


KERAS 3 SAVE FORMAT OPTIONS
============================

1. HDF5 Format (.h5) - BACKWARD COMPATIBLE
   Usage: model.save('model.h5', save_format='h5')
   Pros:
   - Compatible with Keras 2.x
   - Single file contains everything
   - Good for model versioning
   - Fast loading
   Cons:
   - Requires explicit format parameter in Keras 3
   - Larger file size
   - Less portable across frameworks

2. Keras 3 Native Format (.keras) - RECOMMENDED BY KERAS TEAM
   Usage: model.save('model.keras')
   Pros:
   - Default format in Keras 3
   - Better cross-framework support
   - More efficient storage
   - No format parameter needed
   Cons:
   - Not compatible with Keras 2.x
   - Requires code changes to use .keras extension

3. TensorFlow SavedModel Format
   Usage: model.save('saved_model/', save_format='tf')
   Pros:
   - Framework-agnostic
   - TensorFlow Serving compatible
   - Production-ready
   Cons:
   - Directory-based (not single file)
   - Larger size
   - More complex structure


WHY WE CHOSE .h5 FORMAT
=======================

Reasons for keeping .h5 format:
1. Backward compatibility - existing code expects .h5 files
2. Single file storage - easier to manage than directory-based
3. Model naming - code uses {symbol}_lstm_model.h5 pattern
4. Path expectations - multiple places in code reference .h5 extension
5. Minimal code changes - only need to add save_format parameter

Alternative (future consideration):
- Migrate to .keras format for better Keras 3 support
- Would require updating all model path references
- More efficient storage and loading
- Better for long-term maintenance
