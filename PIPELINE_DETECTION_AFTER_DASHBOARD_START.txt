**Detection Method** (_check_for_updated_reports, lines 476-518):
```python
def _check_for_updated_reports(self) -> bool:
    now = datetime.now()
    
    # Only check every 30 minutes
    if now - self._reports_last_check < self._reports_check_interval:
        return False
    
    self._reports_last_check = now
    
    report_base = Path(__file__).parent.parent / 'reports' / 'screening'
    updated = False
    
    for market in ['au', 'us', 'uk']:
        report_path = report_base / f'{market}_morning_report.json'
        if report_path.exists():
            # Check file modification time
            file_mtime = datetime.fromtimestamp(report_path.stat().st_mtime)
            
            # If report exists in cache, check if file is newer
            if market in self._overnight_reports_cache:
                cached_report = self._overnight_reports_cache[market]
                if 'timestamp' in cached_report:
                    cached_time = datetime.fromisoformat(cached_report['timestamp'])
                    if file_mtime > cached_time:
                        logger.info(f"[PIPELINE] Detected updated {market.upper()} morning report")
                        updated = True
            else:
                # New report that wasn't there before
                logger.info(f"[PIPELINE] Detected new {market.upper()} morning report")
                updated = True
    
    if updated:
        logger.info("[PIPELINE] Reloading morning reports...")
        self._overnight_reports_cache = self._load_overnight_reports()
        return True
    
    return False
```

KEY MECHANISMS:
---------------
1. **File Modification Time (mtime)**: Checks OS-level file modification timestamp
2. **Cached Timestamp**: Compares against timestamp IN the JSON report
3. **New File Detection**: Detects reports that didn't exist before
4. **30-Minute Check Interval**: Balances responsiveness vs system load

TIMING DETAILS
================================================================================

TRADING CYCLE FREQUENCY:
------------------------
- Dashboard runs trading cycles every **60 seconds**
- Each cycle takes ~5-30 seconds depending on activity
- Continuous loop while dashboard is running

PIPELINE CHECK FREQUENCY:
-------------------------
- Pipeline report check runs every **30 minutes**
- Even though trading cycles run every 60s, the pipeline check
  only happens once every 30 minutes to avoid excessive I/O
- Controlled by: self._reports_check_interval = timedelta(minutes=30)

TIMELINE EXAMPLE:
-----------------
00:00 - Dashboard starts, pipeline check happens
00:01 - Trading cycle (no pipeline check)
00:02 - Trading cycle (no pipeline check)
...
00:29 - Trading cycle (no pipeline check)
00:30 - Trading cycle + PIPELINE CHECK ← Detects new reports!
00:31 - Trading cycle (no pipeline check)
...
01:00 - Trading cycle + PIPELINE CHECK ← Detects new reports!

WHY 30 MINUTES?
---------------
1. **Balance**: Responsive enough to catch overnight pipelines
2. **Efficiency**: Avoids checking file system every 60 seconds
3. **System Load**: Reduces I/O operations
4. **Pipeline Timing**: Overnight pipelines typically run once daily
5. **Intraday Updates**: 30 min is acceptable delay for swing trading

================================================================================
WHAT HAPPENS WHEN NEW REPORT IS DETECTED
================================================================================

STEP-BY-STEP PROCESS:
---------------------
1. **Detection**: _check_for_updated_reports() returns True
   
2. **Logging**: "[PIPELINE] Processing fresh pipeline recommendations..."
   
3. **Extraction**: _get_pipeline_recommendations() called
   - Retrieves top 5 recommendations per market
   - Extracts: symbol, signal, opportunity_score, sentiment, market
   - Creates unique key: "{symbol}_{timestamp}"
   - Filters already processed recommendations
   
4. **Evaluation**: For each recommendation:
   - Check if symbol already has position (skip if yes)
   - Check if max_total_positions reached (stop if yes)
   - Evaluate with _evaluate_pipeline_recommendation()
   - Apply trading criteria and risk management
   
5. **Execution**: For actionable recommendations:
   - Log: "[PIPELINE] Actionable: {symbol} ({market}) - {signal} @ score={score}"
   - Fetch current price with yfinance
   - Call enter_position() to execute BUY order
   - Mark recommendation as processed (avoid duplicates)
   
6. **Tracking**: Add recommendation_key to _processed_recommendations set

7. **Summary**: Log total actionable recommendations processed

================================================================================
GUARANTEES & EDGE CASES
================================================================================

✅ GUARANTEED BEHAVIORS:
-----------------------
1. New pipeline reports WILL BE detected (within 30 minutes)
2. Updated pipeline reports WILL BE detected (within 30 minutes)
3. Multiple simultaneous updates WILL ALL be detected in one check
4. Processed recommendations WILL NOT be duplicated
5. Detection works regardless of when dashboard was started
6. Detection continues for entire dashboard runtime

⚠️ EDGE CASES:
--------------
1. **Very Fast Re-runs**: If pipeline runs twice within 30 minutes,
   only the latest run will be processed (not both)
   
2. **File System Lag**: If file system mtime is delayed/cached,
   detection may be slightly delayed
   
3. **Clock Skew**: If system clocks differ between pipeline server
   and dashboard server, timestamps may not align correctly
   
4. **Race Condition**: If pipeline writes report while coordinator
   is reading it, may get incomplete data (rare, JSON write is atomic)
   
5. **Missing Timestamp**: If JSON lacks 'timestamp' field, falls back
   to file mtime only (less reliable but still works)

🔧 MITIGATION:
--------------
- JSON writes are typically atomic (OS handles this)
- File mtime is OS-level, reliable
- Duplicate prevention via recommendation_key tracking
- Error handling logs warnings but continues
